ASLR (Address Space Layout Randomization)

ASLR is a security feature that randomizes memory addresses used by a program (like stack, heap, libraries) each time it runs.

Disbling ASLR for Local Binary Analysis

You can disable ASLR locally when analyzing a binary, but you won’t be able to rely on a hardcoded address like the one output by GDB's print function.

Disabling ASLR is mainly for convenience and consistency during debugging or exploit development — not necessarily to find final, real-world addresses.

Benefits

  • Predictable addresses make it easier to inspect and interact with code.

  • If you're developing an exploit (like a buffer overflow to jump to win()), you want consistent testing while crafting it.

  • You can re-run the binary repeatedly and get the same memory layout every time.

Challenges: The target function's address would likely be different across-

  • Different runs

  • Different systems

  • Even different terminal tabs

You’d need to use leaks (like format string or infoleak vulnerabilities) to dynamically find the actual address at runtime.

Use Case

  • Understand the binary's structure.

  • Practice and test your exploit reliably.

  • See if your payload would work in ideal (non-randomized) conditions.

Then, once you’ve got it working, you can:

  • Tackle ASLR separately (e.g. by finding a leak).

  • Use techniques like ROP, dynamic resolution (e.g. ret2plt), or brute force if needed.

DEP (Data Execution Prevention)

DEP is a set of hardware and software technologies that perform additional checks on memory to help protect against malicious code exploits.

Hardware-enforced DEP marks all memory locations in a process as non-executable unless the location explicitly contains executable code.
