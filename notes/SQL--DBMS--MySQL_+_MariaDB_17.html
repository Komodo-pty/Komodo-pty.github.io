Extract Current Queries (Clear-Text Creds)

[CTF D 4-5]

If you can't crack the hashes stored in a DB, you may be able to extract the currently running queries (i.e. login attempts from all users) from the built-in table information_schema.processlist.

Depending on the hashing implementation, you may be able to recover clear-text creds before the password is hashed. You can filter the binary data to only return relevant information.

Hashing Implementation

If the creds are passed to a MySQL hashing function (e.g. MD5() ), you'll be able to capture clear-text creds.

However, if the password hashing is done via the Web App's PHP code, then the password will be hashed before it's passed to the MYSQL query & this Attack Vector won't work.

Background

Schema

https://dev.mysql.com/doc/refman/8.0/en/information-schema-table-reference.html

If you can't find the information_schema (or equivilant), there may be an alternate name (e.g for certain versions).

Any version could be using mysql.innodb_table_stats instead. MySQL didn't introduce information_schema until version 5.0

Reserved Words

In MySQL, certain keywords are “reserved” (e.g. the names of built-in functions) & require special treatment when being used. There are also some non-reserved words that are still somehow restricted.

In some cases, the group Column can tell you which users have elevated privs (e.g. Admin privs), letting you ID high value targets. However, group is a reserved word that needs to be escaped.

If you don't escape a reserved word, you may get an error like access violation

Escaping Reserved Words

When using these words as an identifier (e.g. Table or Columns name), you need to escape them with Backticks ( ` ).

bob" UNION ALL SELECT NULL,`group` FROM web.users-- //

Primary Key Constraint

The Primary Key is a unique identifier for each row in a table. It also implicitly creats a unique index for that Column or set of Columns, ensuring that there are no duplicate values for the Primary Key.

For example, multiple users could have the same value for the group Column, but they can't have the same id. Therefore, the Primary Key Constraint would prevent any users from sharing the same value for the id Column.

Comparing Values (LIKE)

[CTF D 5-6]

The = & != operators treat the values literally, whereas LIKE & NOT LIKE operators support REGEX (i.e. Wildcards like % any 0+ values & _ any 1 value).

Binary Data

The LIKE operator is designed to work with textual data (e.g. UTF-8, Latin1, etc), where LIKE will look at the characters in the string & pattern match accordingly.

When used on Binary Columns (e.g. BLOB, VARBINARY, or INFO_BINARY), the LIKE operator treats the data as a sequence of raw bytes.

In this case, the comparison is done byte-to-byte without interpretting characters, encodings, or collations (the sort order for SQL), & each byte is treated as a literal value instead of as a part of a character set.

MySQL & LIKE treat binary data as a string of bytes. As a result, MySQL will look for any part of the binary data in INFO_BINARY (for example) that matches the exact byte sequence representing your filter value.

Alternatively, the binary data could contain some textual data (i.e. the string you're using for the filter value). This is similar to how you can view some words when running cat binary_file.bin.

Note: This kind of filtering may be less reliable when working with truly arbitrary bytes (purely non-text). You may want to use an alternate method to properly interpret / filter the binary data (e.g. using HEX() to view the data in hexadecimal form) .

Example Query: I think that your filter string (INFO_BINARY) is converted into binary, & queries which contain that same binary are excluded.

SELECT INFO_BINARY FROM information_schema.processlist WHERE INFO_BINARY NOT LIKE “%INFO_BINARY%” LIMIT 1
